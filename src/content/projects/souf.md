    cause:
      - "CloudWatch CPUUtilization 지표로 단일 인스턴스의 CPU가 한계에 도달한 것을 확인했고, 요청 폭증에 대응할 수 있는 확장 구조가 없어 처리 한계를 초과했다고 판단했습니다."

      - "CPU 사용률 70% 초과 시 ASG scale-out 정책 적용"
    result:
      - "이벤트 진행 중 활성 사용자 9명 -> 1100명(약 220%)으로 증가하는 상황에서도 평균 응답 시간 정상 유지"
      - "서버 장애 0건"
    lesson:
      - "비용 제약 안에서도 모니터링 임계치와 자동 확장 정책으로 안정성을 확보할 수 있음을 학습"
      - "이벤트 전 부하 테스트로 병목 지점을 사전에 파악해야 한다는 필요성 확인"
  - key: "Caching"
    title: "인기 게시글 반복 조회를 캐싱으로 응답 속도 안정화"
    image: "/projects/souf-problem.svg"
    imageAlt: "인기 게시글 조회 요청이 데이터베이스에 반복 접근하는 문제"
    problem:
      - "메인 페이지의 인기 게시글은 다수의 사용자가 동시에 반복 조회하는 API였고, 요청이 집중되며 DB 커넥션 풀이 고갈되어 다른 API 요청까지 연쇄적으로 지연되는 문제가 발생했습니다."
    cause:
      - "쓰기가 거의 발생하지 않는 인기 게시글 데이터임에도 매 요청마다 DB에 직접 접근하는 구조였기 때문에 불필요한 부하가 누적되었습니다. 평균 응답 시간은 약 312ms로 측정되었고, 트래픽이 몰릴수록 커넥션 대기가 쌓이는 구조적 한계를 확인했습니다."
    solution:
      - "읽기 요청이 압도적으로 많은 API 특성을 기준으로 Look-aside(Cache-aside) 캐싱 전략을 채택했습니다. 인기 게시글은 메인 페이지 노출 특성상 동시 요청 수가 집중되므로, 매 요청이 DB 커넥션을 점유하면 전체 서비스의 DB 커넥션까지 블로킹될 위험이 있었습니다."
      - "Write-through 전략도 검토했지만, 데이터를 쓸 때 캐시와 DB를 동시에 갱신하는 방식은 쓰기가 빈번한 데이터에 더 적합했습니다. 인기 게시글처럼 쓰기가 거의 없는 데이터에는 불필요한 오버헤드가 크다고 판단했습니다."
      - "캐시 정합성을 위해 스케줄러를 적용해 특정 시간대에 DB에서 인기 게시글을 갱신하도록 구성했고, 오래된 캐시 데이터가 계속 서빙되는 문제를 방지했습니다."
      - "Redis 기반 Look-aside(Cache-aside) 캐싱 적용"
    result:
      - "평균 응답 시간 312ms → 57ms, 81.7% 단축"
      - "DB 커넥션 풀 고갈 문제 해소"
      - "메인 페이지 응답 지연 방지"
    lesson:
      - "읽기/쓰기 패턴을 기준으로 캐싱 전략을 선택해야 한다는 점을 학습"
      - "단일 API 병목이 커넥션 풀을 통해 전체 서비스 장애로 확산될 수 있음을 확인"
  - key: "Distributed Lock"
    title: "분산락을 이용한 스케줄러 중복 실행 방지"
    image: "/projects/souf-tech.svg"
    imageAlt: "여러 서버 인스턴스에서 스케줄러 중복 실행을 막는 분산락 구조"
    problem:
      - "AWS ALB + ASG로 다중 인스턴스가 운영되는 환경에서, 인기 게시글 업데이트 스케줄러가 각 인스턴스마다 동시에 실행되는 중복 실행 문제가 발생하였습니다."
      - "중복 실행을 방치할 경우, 동일한 데이터를 여러 인스턴스가 동시에 읽고 쓰면서 데이터 정합성이 오염될 가능성이 있으며, DB 부하가 가중이 되는 결과가 이어질 가능성이 매우 높았습니다."
    cause:
      - "해당 문제는 각 서버 인스턴스는 독립적인 프로세스로 동작하기 때문에, 별도의 락을 걸어주지 않는다면 다른 인스턴스의 실행 여부를 알 수 없는 것이 원인이었습니다."
    solution:
      - "문제를 해결하기 위해 저는 Redis 기반 분산락을 선택하기로 했습니다. 하나의 Redis 인스턴스에서 락을 관리하기 때문에 DB 부하 없이 인스턴스 간 락 공유가 가능하며, 추가로 TTL(Time-To-Live) 설정으로 장애 시 자동 해제를 보장할 수 있다는 것이 제 판단이었습니다."
    result:
      - "다중 인스턴스 환경에서 스케줄러가 단일 인스턴스에서만 실행되는 것을 보장하였고, DB 중복 쿼리 및 캐시 불일치 문제를 해소할 수 있었습니다."
      - "분산락 적용 전 로그에 동일 작업이 3번 찍혔으나, 적용 후에 로그에 1번만 찍히는 것을 확인할 수 있었습니다."
      - "ASG 환경에서도 스케줄러 단일 실행 보장하였습니다."
    lesson:
      - "정량적 수치는 측정하지 못했으나, 로그 기반으로 단일 실행을 확인하였고 향후에 모니터링 지표 설정의 필요성을 느꼈습니다."
      - "단일 서버 환경에서 정상적으로 동작하던 코드가 다중 인스턴스 환경에서는 전혀 다른 문제를 일으킬 수 있다는 것을 경험했습니다. 확장성을 고려한 설계가 단순히 성능만의 문제가 아니라 데이터 정합성과 직결된다는 점, 그리고 락 전략 선택 시 환경과 트레이드오프를 함께 고려해야 한다는 것을 배웠습니다."
  - key: "Asynchronous Pipeline"
    title: "비동기 썸네일 추출 파이프라인을 구성했습니다."
    image: "/projects/souf-solution.svg"
    imageAlt: "EventListener, SQS, ECS 기반 비동기 썸네일 처리 구조"
    problem:
      - "동영상을 포함한 피드가 있을 경우, 피드 리스트 조회 시 동영상 원본 파일을 직접 불러오는 과정에서 전체 리스트 응답이 지연되는 문제 발생. 동영상이 포함된 피드가 하나라도 있으면 나머지 피드들의 응답까지 블로킹되어 사용자 경험 저하."
    cause:
      - "피드 리스트 조회 시 S3에 저장된 동영상 원본(최대 500MB)의 URL을 반환하고, 클라이언트가 해당 URL로 동영상을 직접 로딩하는 구조. 썸네일 없이 대용량 동영상을 참조하기 때문에 클라이언트 렌더링 단계에서 지연이 발생하고, 동영상이 포함된 피드가 많을수록 사용자가 체감하는 로딩 속도가 크게 저하됨."
    solution:
      - "동영상 원본 대신 경량 썸네일 이미지를 미리 추출해두는 비동기 파이프라인 구성."
      - "FFmpeg 컨테이너 이미지화: 오픈소스 FFmpeg 라이브러리를 Docker 이미지로 빌드하여 ECR에 저장, 썸네일 추출 작업을 독립적인 컨테이너로 분리"
      - "이벤트 기반 트리거: 피드 생성/수정 시 동영상 업로드가 발생하면 EventListener를 통해 AWS SQS로 메시지 전송"
      - "비동기 썸네일 추출: SQS 메시지를 트리거로 ECS가 FFmpeg 컨테이너를 실행하여 썸네일 추출 작업 수행. 업로드 응답과 완전히 분리된 비동기 처리로 API 응답 지연 제거"
      - "피드 조회 개선: 이후 피드 리스트 조회 시 동영상 원본 대신 추출된 썸네일 이미지를 반환하도록 변경"
    result:
      - "피드 리스트 조회 시 동영상 원본 파일 참조를 완전히 제거하여 응답 시간 대폭 단축."
      - "동영상 포함 피드가 다수 존재하더라도 리스트 응답이 블로킹되지 않음"
    lesson:
      - "대용량 파일 처리는 사용자 요청 흐름과 분리해야 장애 전파와 체감 지연을 줄일 수 있다는 점을 배웠습니다."
      - "비동기 파이프라인은 단순 성능 개선뿐 아니라 실패 격리와 운영 안정성에도 중요한 설계 선택임을 확인했습니다."
  - key: "Reduce Bounce Rate"
    title: "행동 데이터 기반 UI/UX 개선으로 이탈률 감소 및 회원 확보"
    image: "/projects/souf-result.svg"
    imageAlt: "Google Analytics 경로 탐색 분석과 이탈률 개선 결과"
    problem:
      - "Google Analytics를 통해 메인 페이지와 회원가입 페이지에서 사용자 이탈이 집중적으로 발생하는 것을 확인. 피드 이벤트 운영 중임에도 활성 사용자가 9명에 그쳐 플랫폼 성장에 심각한 병목으로 작용."
    cause:
      - "게릴라 테스트(Guerrilla Testing)를 통해 두 가지 핵심 원인을 파악. 첫째, 메인 페이지에 플랫폼의 핵심 가치를 전달하는 콘텐츠가 부족하여 사용자가 서비스 이용의 필요성을 느끼지 못하고 이탈. 둘째, 회원가입이 단일 페이지에 집중된 복잡한 입력 폼으로 구성되어 진행 상황을 파악하기 어렵고 이탈률이 높았음. 분석 결과를 JIRA Confluence에 정리하여 팀 전체와 공유하고 개선 방향 수립."
    solution:
      - "메인 페이지와 회원가입 두 영역을 분리하여 개선."
      - "메인 페이지 개선 랜딩 화면 상단에 핵심 콘텐츠를 노출하도록 UI 개편하여 사용자가 서비스 가치를 즉시 인지할 수 있도록 구조 변경."
      - "회원가입 플로우 재설계 (백엔드) 단일 폼을 3단계 스텝 구조로 분리하여 사용자가 진행 상황을 명확히 인지하고 완료 기대감을 유지하도록 재설계. 단계별로 유효성 검사 범위를 분리함으로써 API 오류 발생 시 원인 파악과 수정이 용이한 구조로 개선."
      - "소셜 로그인을 병행 도입하여 입력 부담을 최소화. 회원가입 유형(일반/소셜)별 데이터를 sealed class로 명확히 분리하여 컴파일 타임 타입 안전성 확보 및 분기 처리 누락 방지."
    result:
      - "메인 페이지 이탈률 31% → 5% 로 개선."
      - "회원가입 사용자 30명 → 238명 으로 약 693% 증가."
    lesson:
      - "기능 구현 중심의 개발에서 벗어나 Google Analytics 데이터를 직접 분석하고 사용자 행동 기반으로 개선 방향을 도출하는 경험을 했습니다. 백엔드 개발자도 프론트엔드 UX 흐름과 데이터 분석에 대한 기본적인 이해가 있어야 더 나은 API 설계와 서비스 개선이 가능하다는 것을 배웠습니다."
