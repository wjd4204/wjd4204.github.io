      요청이 들어올 때마다 DB에서 주간 조회 수를 기준으로 상위 6개의 피드를 조회했고, 이 과정에서 DB 커넥션 풀이 지속적으로 사용되었습니다."
    cause:
      - "문제의 핵심은 인기 피드 데이터의 특성과 조회 방식이 어울리지 않는 것에 있었습니다."
      - "인기 피드는 매번 바뀌어야 하는 데이터가 아니라 일정 주기마다 갱신되어도 서비스 품질에 큰 영향을 주지 않는 데이터였습니다.
      메인 페이지 접근과 함께 DB 접근이 반복적으로 발생하도록 설계한 것이 원인이었습니다."
      - "이런 방식이 지속될 경우, 요청 수가 증가할수록 DB 커넥션 사용량도 함께 증가하는 문제를 초래하고,\n
      **UX를 크게 저하**시킬 수 있는 구조였습니다."
      - "따라서 단순히 쿼리를 개선하는 것보다, 반복적인 DB 접근 자체를 줄이는 캐싱 전략이 더 적합하다고 판단하게 되었습니다."
    solution:
      - "인기 피드는 Redis 자료구조를 직접 다루기보다 주간 조회 수 기준 Top6 결과를 저장하고 반환하는 조회 결과 캐싱에 가까웠습니다.\n
      그래서 RedisTemplate로 명령어를 직접 다루기보다 Spring Cache 추상화와 Redis 저장소를 조합해 캐싱 의도를 코드에서 명확히 드러냈습니다."
      - "**Google Analytics** 탐색 기능으로 사용자 접속률이 가장 낮은 시간이 매주 월요일 자정대인 것을 확인했고,\n
      매주 월요일 00시 5분에 스케줄러가 인기 피드를 집계하고 Redis 캐시를 미리 갱신하는 **Refresh-ahead 전략**을 주전략으로 적용했습니다."
      - "서버 재시작이나 캐시 미스처럼 캐시가 비어 있는 예외 상황에 대비해 **Look-aside를 fallback**으로 구성했습니다."
      - "일반 상황에서는 Redis 캐시를 바로 반환하고, 캐시가 없을 때만 DB 조회 후 Redis에 다시 저장하도록 만들어 DB 커넥션 풀 부담을 줄였습니다."
    result:
      - "평균 응답 시간 512ms → 57ms, 약 88.9% 단축"
      - "메인 페이지 응답 지연 문제를 해결"
      - "사용자가 처음 접하게 되는 페이지의 문제를 조기에 발견하여 UX 개선"
    lesson:
      - "캐싱은 Redis를 도입하는 것보다, 먼저 **데이터의 조회 패턴과 갱신 주기**를 이해하는 일이 중요하다는 것을 배웠습니다."
      - "인기 피드처럼 정해진 시점에 집계되는 조회 결과는 Spring Cache 추상화로 캐싱 의도를 드러내고, Redis는 다중 인스턴스가 공유하는 외부 저장소로 두는 방식이 적합하다는 것을 배웠습니다."
      - "Look-aside fallback 전략을 추가 구성하며, 캐시가 없을 때도 DB 조회 후 Redis를 다시 채우는 **복구 경로를 반드시 설계**해야 한다는 점을 경험했습니다."
    troubleshootingSummary:
      problem: "인기 피드가 페이지 진입마다 DB를 조회해 커넥션 풀 부담이 커졌습니다."
      judgment: "데이터 성격상 실시간성보다 반복 조회 비용을 줄이는 캐싱이 더 적합했습니다."
      result: "평균 응답 시간을 512ms에서 57ms로 줄였습니다."

    solution:
      - "동영상 원본을 피드 리스트 응답 경로에서 제거하기 위해, 업로드 시점에 경량 썸네일을 미리 추출하는 **비동기 파이프라인**을 구축했습니다."
      - "썸네일 추출 도구는 **FFmpeg**를 선택했습니다. AWS Elastic Transcoder나 MediaConvert는 단순 썸네일 추출 목적에는 비용과 설정 부담이 크다고 판단했습니다."
      - "FFmpeg를 Docker 이미지로 빌드해 AWS ECR에 저장하고, 썸네일 추출 작업을 애플리케이션 서버와 분리된 **컨테이너 작업으로 구성**했습니다."
      - "피드 생성 및 수정 시 동영상 업로드가 발생하면 EventListener가 AWS SQS로 메시지를 전송하고, 업로드 응답은 메시지 전송 직후 반환하도록 분리했습니다."
      - "SQS 메시지를 트리거로 ECS가 FFmpeg 컨테이너를 실행해 썸네일을 추출하고, 결과 파일을 S3의 지정 경로에 저장하도록 구성했습니다."
      - "결과적으로 피드 리스트 조회 시 동영상 원본 URL 대신 썸네일 이미지 URL을 반환하도록 변경해 **리스트 렌더링 지연을 줄일 수 있었습니다.**"
    result:
      - "500MB에 달하는 동영상 원본으로 테스트해본 결과, 렌더링 경로에서 완전히 제거되어 **지연 원인이 차단됨을 확인**"
      - "동영상 업로드 시 응답이 썸네일 추출 작업과 분리되어 사용자는 업로드 완료 즉시 서비스 이용 가능"
    lesson:
      - "무거운 작업을 API 응답 경로에서 분리하는 것만으로도 사용자 체감 성능이 크게 개선될 수 있다는 것을 경험했습니다."
      - "**이벤트 기반 비동기 파이프라인**을 설계하며, 각 컴포넌트가 약한 의존성을 지닐수록 전체 서비스의 흐름에 지연이 생기지 않는다는 것을 배웠습니다."
    troubleshootingSummary:
      problem: "동영상 원본 URL이 피드 리스트에 섞이며 렌더링이 느려졌습니다."
      judgment: "렌더링 지연을 해소하기 위해 경량화된 동영상 썸네일을 생성해야겠다고 판단했습니다."
      result: "비동기 썸네일 추출 파이프라인을 구축하여 렌더링 지연을 해소할 수 있었습니다."
  - key: "Reduce Bounce Rate"
    title: "행동 데이터 기반 UI/UX 개선으로 이탈률 감소 및 서비스 활성화"
    image: "/projects/souf-google-analytics.png"
    imageAlt: "Google Analytics 기반 Before After 사용자 지표 개선 결과"
    problem:
      - "Google Analytics를 통해 메인 페이지와 회원가입 페이지에서 사용자 이탈이 집중적으로 발생하는 것을 확인했습니다.
      활성 사용자가 9명에 그쳐 플랫폼이 실질적으로 성장하지 못하고 있었습니다."
    cause:
      - "Google Analytics 유입 채널 분석 결과와 주변 지인을 대상으로 한 **게릴라 테스트**를 바탕으로, 두 가지 핵심 원인을 파악했습니다."
      - "1. Instagram이 전체 유입의 약 48%를 차지했으나 메인 페이지에 플랫폼의 핵심 가치를 전달하는 **컨텐츠가 부족**하여 서비스 이용의 필요성을 느끼지 못하고 이탈하는 것을 확인했습니다."
      - "2. 회원가입이 단일 페이지에 집중된 **복잡한 입력 폼으로 구성**되어 사용자들이 어려움을 느껴 회원가입을 중도 포기하는 것을 확인했습니다."
      - "조사 결과를 기반으로 분석 결과와 개선 방향을 JIRA Confluence에 문서화하여 팀 전체와 공유했습니다.
      각 개선 항목은 JIRA 이슈로 등록하여 우선순위를 정하고 작업을 분배하는 방식으로 체계적으로 대응했습니다."
    solution:
      - "메인 페이지와 회원가입 플로우를 분리해 각각의 **이탈 원인에 맞게 개선**했습니다."
      - "메인 페이지 상단에 **서비스 핵심 가치를 먼저 노출하도록 랜딩 구조를 개편**해, 사용자가 진입 직후 서비스 이용 필요성을 이해할 수 있게 했습니다."
      - "회원가입은 단일 폼을 **3단계 스텝 구조**로 나누어 사용자가 진행 상황을 확인하면서 입력할 수 있도록 재설계하여 
      단계별 유효성 검사 범위를 분리해 API 오류가 발생했을 때 원인을 더 빠르게 파악하고 수정할 수 있게 했습니다."
      - "추가적으로 **소셜 로그인을 병행 도입**해 입력 과정을 줄이고, 가입 중간 이탈 가능성을 낮췄습니다."
      - "회원가입 유형은 기존 enum 타입에서 sealed class로 정리해 유형별 필요한 데이터를 분리하고 분기 처리 누락을 줄였습니다."
    result:
      - "메인 페이지 이탈률을 **37% -> 5%**로 개선하였습니다."
      - "개선 이후 이벤트 기간 진행 중 활성 사용자가 **9명 -> 1,100명**으로 증가하였습니다."
    lesson:
      - "사용자 행동 기반으로 개선 방향을 도출하며 백엔드 개발자도 프론트엔드 **UX 흐름과 데이터 분석에 대한 기본적인 이해**가 있어야 더 나은 API 설계와 서비스 개선이 가능하다는 것을 배웠습니다."
      - "JIRA를 통해 분석 결과를 이슈로 체계화하고 팀과 공유하는 과정에서 개발 외적인 협업 프로세스가 문제 해결의 속도와 완성도에 직접적인 영향을 미친다는 것을 경험했습니다."
    troubleshootingSummary:

      judgment: "Google Analytics 데이터와 사용자 테스트를 바탕으로, 서비스 가치 전달 구조와 회원가입 플로우를 우선 개선했습니다."
      result: "메인 이탈률을 37%에서 5%로 낮추고 활성 사용자를 1,100명까지 늘렸습니다."
