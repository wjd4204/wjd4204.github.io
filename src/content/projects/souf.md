    cause:

      - "1. 확장 한계와 SPOF 문제 발생\n 수직 확장은 해당 인스턴스에서 장애가 발생하면 서비스 전체가 중단됩니다.\n 피드 이벤트처럼 트래픽이 단기간에 집중되는 서비스 특성상 해당 리스크는 감수하기 어려웠습니다."
      - "2. 비용 탄력성\n 수직 확장은 큰 인스턴스를 상시 유지해야하기 때문에 트래픽이 없는 평상시에도 고정 비용이 발생합니다. 반면 수평 확장은 평소에 최소 인스턴스로 운영하다가 트래픽 급증 시에만 자동으로 확장되어 시간을 기반으로 한 비용 효율이 높다고 판단하였습니다."
      - "이에 따라 ALB로 요청을 분산하고, ASG로 인스턴스를 최대 3대까지 자동 확장하는 3-tier 구조로 개선했습니다.\n scale-out 트리거 기준을 CPU 사용률
      70%로 설정하였는데 그 이유는 새 인스턴스의 warm start 대기 시간을 고려했기 때문입니다. CPU가 100%에 도달한 뒤 scale-out을 트리거하면,
      새 인스턴스가 부팅을 완료하고 ALB 헬스체크를 통과하여 실제 트래픽을 받기까지의 20초 동안 기존 인스턴스가 단독으로 과부하를 감당해야 했습니다.
      이를 방지하기 위해 70% 시점에 선제적으로 확장을 트리거하여 워밍업 대기 시간을 확보했습니다."

      - "사전에 scale-out 임계치를 설정해두는 자동화된 대응 구조가 이벤트 기반 서비스에서 얼마나 중요한지 직접 경험할 수 있었습니다."

  - key: "Caching"
    title: "캐싱 병행 전략을 통한 응답 속도 안정화"
    image: "/projects/souf-redis-strategy.png"
    imageAlt: "Refresh-ahead와 Look-aside를 함께 사용하는 Redis 캐싱 전략"
    problem:
      - "메인 페이지의 인기 피드 리스트가 상시 요청되면서 인기 피드 API가 계속 DB를 직접 조회했습니다.\n
      인기 피드는 메인 페이지에 항상 노출되기 때문에 사용자가 페이지에 접근할 때마다 조회 요청이 발생했습니다.\n
      요청이 들어올 때마다 DB에서 주간 조회 수를 기준으로 상위 6개의 피드를 조회했고, 이 과정에서 DB 커넥션 풀이 지속적으로 사용되었습니다."
    cause:
      - "문제의 핵심은 인기 피드 데이터의 특성과 조회 방식이 어울리지 않는 것에 있었습니다."
      - "인기 피드는 매번 바뀌어야 하는 데이터가 아니라 일정 주기마다 갱신되어도 서비스 품질에 큰 영향을 주지 않는 데이터였습니다.
      메인 페이지 접근과 함께 DB 접근이 반복적으로 발생하도록 설계한 것이 원인이었습니다."
      - "이런 방식이 지속될 경우, 요청 수가 증가할수록 DB 커넥션 사용량도 함께 증가하는 문제를 초래하고,\n
      UX를 크게 저하시킬 수 있는 구조였습니다."
      - "따라서 단순히 쿼리를 개선하는 것보다, 반복적인 DB 접근 자체를 줄이는 캐싱 전략이 더 적합하다고 판단하게 되었습니다."
    solution:
      - "Redis를 직접 제어하는 방식보다, Spring Cache 추상화를 사용하고 실제 저장소는 Redis로 구성하는 방식을 선택하였습니다."
      - "Spring Cache를 선택한 이유는 인기 피드 데이터가 자료구조를 직접 다루는 데이터가 아닌 주간 조회 수 기준으로 집계된 Top6 결과를 저장하고 반환하는
      단순 조회 결과 캐싱에 가까웠기 때문입니다. 따라서 RedisTemplate로 Redis 명령어를 직접 다루기보다 CacheManager를 사용해
      캐싱 의도를 명확히 드러내는 방식이 더 적합하다고 판단했습니다."
      - "이후 인기 피드을 언제 갱신할지 결정했습니다.\n
      Google Analytics 탐색 기능을 통해 사용자 접속률이 가장 낮은 시간이 매주 월요일 자정대임을 확인했습니다.
      그래서 매주 월요일 00시 5분에 스케줄러가 인기 피드를 집계하고 Redis 캐시를 갱신하도록 설계했습니다."
      - "캐싱 전략은 Refresh-ahead를 주전략으로 선택했습니다.\n
      인기 피드은 갱신 시점 외에는 대부분 읽기 요청만 발생하는 데이터였기 때문에, 사용자가 요청한 뒤 캐시를 채우는 방식보다
      스케줄러가 미리 캐시를 갱신해두는 방식이 더 적합했습니다. 다만 Refresh-ahead만으로는 서버 재시작이나 캐시 미스와 같은 예외 상황을 모두 처리하기 어렵다고 판단했습니다.
      캐시가 비어있는 상황에서도 서비스가 정상적으로 운영되어야 했기 때문에 Look-aside를 fallback 전략으로 함께 구성했습니다."
      - "결과적으로 일반적인 상황에서는 스케줄러가 미리 갱신한 Redis 캐시를 통해 DB 접근 없이 인기 피드를 반환하고, 
      캐시가 비어 있는 예외 상황에서는 DB에서 데이터를 조회한 뒤 다시 Redis에 저장하도록 만들었습니다.\n
      이를 통해 DB 커넥션 풀 부담을 줄이면서도, 캐시 유실 상황에 대응할 수 있는 인기 피드 갱신 파이프라인 구조를 만들었습니다."
    result:
      - "평균 응답 시간 512ms → 57ms, 약 88.9% 단축"
      - "메인 페이지 응답 지연 문제를 해결"
      - "사용자가 처음 접하게 되는 페이지의 문제를 조기에 발견하여 UX 개선"
    lesson:
      - "캐싱 전략은 단순히 Redis를 도입하는 문제가 아니라, 데이터의 목적과 갱신 주기를 먼저 이해해야 한다는 점을 배웠습니다."
      - "단순 조회 결과 캐싱에는 Spring Cache 추상화를 활용하고, 다중 인스턴스 환경에서는 Redis를 외부 캐시 저장소로 두는 방식을 적용하며
      Spring Cache와 Redis가 필요한 상황을 명확하게 이해하게 되었습니다."
      - "Refresh-ahead만으로는 캐시 유실 상황을 대처할 수 없다는 문제를 Look-aside를 fallback으로 구성하며, 캐시가 없을 때 어떻게 복구할지 대비 방안을 설계하는 과정에서 어떤 전략이든 예외 상황을 고려해야한다는 것을 경험했습니다."
    troubleshootingSummary:
      problem: "인기 피드가 페이지 진입마다 DB를 조회해 커넥션 풀 부담이 커졌습니다."
      judgment: "데이터 성격상 실시간성보다 반복 조회 비용을 줄이는 캐싱이 더 적합했습니다."
      result: "평균 응답 시간을 512ms에서 57ms로 줄였습니다."
  - key: "Distributed Lock"
    title: "분산락을 이용한 스케줄러 단일 실행 환경 보장"
    image: "/projects/souf-distributed-lock.png"
    imageAlt: "Redis 분산락으로 다중 EC2 스케줄러 중복 실행을 방지하는 구조"
    problem:
      - "앞서 작성한 트러블슈팅에서 구축한 3-tier 아키텍쳐와 Refresh-ahead 파이프라인이 맞물리며 새로운 문제가 발생했습니다."
      - "다중 인스턴스 환경에서 인기 피드 업데이트 스케줄러가 각 인스턴스마다 동작하는 중복 실행 문제였습니다. 
      해당 문제를 검증하기 위해 의도적으로 부하를 발생시켜 인스턴스 2대를 가동시킨 뒤 스케줄러를 실행한 결과, 각각의 인스턴스에서 스케줄러 로그가 찍히는 것을 확인할 수 있었습니다.
      중복 실행을 방치할 경우 동일한 데이터를 여러 인스턴스가 동시에 읽고 쓰면서 캐시 데이터 정합성이 오염될 수 있으며,
      DB에 동일한 집계 쿼리가 인스턴스 수만큼 중복 실행되어 불필요한 DB 부하가 가중될 위험이 있다고 판단했습니다."
    cause:
      - "각 서버 인스턴스는 독립적인 프로세스로 동작하기 때문에, 서로가 스케줄러 동작 여부를 확인할 수 없습니다. 
      결과적으로 단일 서버 환경을 전제로 설계된 스케줄러가 다중 인스턴스에서 치명적인 문제점을 일으킨 것이 원인이었습니다."
    solution:
      - "사전에 Google Analytics 탐색 기능으로 도출된 사용자 접속률이 가장 낮은 매주 월요일 자정에 인기 피드 갱신 스케줄러가 실행되도록 설정했습니다. 
      처음에는 접속률이 낮은 시간대라면 ASG 환경에서도 인스턴스가 여러 대로 유지될 가능성이 현저히 낮아, 스케줄러 중복 실행까지 고려해야 하는지 의문이 있었습니다. 
      하지만 접속률이 낮은 시간대를 선택한 것은 DB 부하를 줄이기 위한 결정이었고, 현재 직면한 문제에 대한 분산락은 다중 인스턴스 환경에서 스케줄러가 중복 실행되지 않도록 보장하기 위한 결정이었습니다. 
      즉, 두 선택은 같은 문제를 해결하는 것이 아니라 서로 다른 위험을 줄이기 위한 결정이었습니다."
      - "락 방식별로 검토한 결과는 아래와 같습니다.\n
      1. 서버 로컬 락은 JVM 내부에서 동작하는 방식이라 다중 인스턴스 환경에서는 적합하지 않았습니다.\n
      2. DB 락은 락 획득과 해제마다 DB 트랜잭션이 일어나게 되어 부하의 증가를 야기할 수 있으며, 트랜잭션의 관리 복잡도가 증가한다는 한계가 존재했습니다.
      이 과정에서 DB 부하를 줄이는 것이 목적인 본 스케줄러의 목적과는 모순되어 선택하지 않게 되었습니다.\n
      3. redisson은 이미 캐싱 용도로 이용 중인 redis의 인프라를 그대로 활용할 수 있어 별도의 라이브러리 없이 도입할 수 있었습니다.
      redis 단일 인스턴스에서 락을 관리하므로 별도의 프로세스로 운영 중인 인스턴스끼리의 락 공유가 가능하며, DB 부하를 최소화할 수 있었습니다.
      또한 TTL 설정을 통해 스케줄러 실행 중 문제가 발생하더라도 락이 자동 해제되어 데드락 상태를 방지할 수 있었습니다."
    result:
      - "다중 인스턴스 환경에서 스케줄러가 단일 인스턴스에서만 실행되는 것을 보장하였고, DB 중복 쿼리 및 캐시 불일치 문제를 해소"
      - "분산락 적용 후 인스턴스 2대 환경에서 스케줄러 실행 로그가 2 -> 1로 감소"
    lesson:
      - "사용자 행동에 대한 통계에 의존한 설계는 예외 상황에 취약할 수 있기 때문에, 코드 레벨의 안전 장치를 병행하는 것이 중요하다는 것을 깨달았습니다."
      - "단일 서버 환경에서 정상적으로 동작하던 코드가 다중 인스턴스 환경에서는 전혀 다른 문제를 일으킬 수 있다는 것을 경험했습니다."
      - "확장성을 고려한 설계가 단순히 성능만의 문제가 아니라 데이터 정합성과 직결된다는 점, 그리고 락 전략 선택 시 환경과 트레이드오프를 함께 고려해야 한다는 것을 배웠습니다."

      - "3. 비동기 썸네일 추출\n

      - "4. 피드 조회 UX 개선\n
      피드 리스트 조회 시 동영상이 포함된 피드의 경우 동영상 원본 URL 대신 썸네일 이미지 URL을 반환하도록 변경했습니다."
