      - "운영 비용을 고려해 단일 인스턴스로 서버를 운영하던 중, 메인 페이지 API 요청 및 DB 커넥트 병목 현상으로 인해 CPU 사용률이 98%까지 급격히 증가하며 서버 응답 불가 상태가 발생했습니다."
    cause:
      - "CloudWatch CPUUtilization 지표로 단일 인스턴스의 CPU가 한계에 도달한 것을 확인했고, 요청 폭증에 대응할 수 있는 확장 구조가 없어 처리 한계를 초과했다고 판단했습니다."
    solution:
      - "초기에는 수직 확장을 고려했습니다.\n 단일 인스턴스를 유지하면 관리 복잡도가 낮고 단기 비용이 적다는 장점이 있었지만,\n 두 가지 이유로 수평 확장을 선택했습니다."
      - "1. 확장 한계와 SPOF 문제 발생\n 수직 확장은 해당 인스턴스에서 장애가 발생하면 서비스 전체가 중단됩니다.\n 피드 이벤트처럼 트래픽이 단기간에 집중되는 서비스 특성상 해당 리스크는 감수하기 어려웠습니다."
      - "2. 비용 탄력성\n 수직 확장은 큰 인스턴스를 상시 유지해야하기 때문에 트래픽이 없는 평상시에도 고정 비용이 발생합니다. 반면 수평 확장은 평소에 최소 인스턴스로 운영하다가 트래픽 급증 시에만 자동으로 확장되어 시간을 기반으로 한 비용 효율이 높다고 판단하였습니다."
      - "이에 따라 ALB로 요청을 분산하고, ASG로 인스턴스를 최대 3대까지 자동 확장하는 3-tier 구조로 개선했습니다.\n scale-out 트리거 기준을 CPU 사용률
      70%로 설정하였는데 그 이유는 새 인스턴스의 warm start 대기 시간을 고려했기 때문입니다. CPU가 100%에 도달한 뒤 scale-out을 트리거하면,
      새 인스턴스가 부탕을 완료하고 ALB 헬스체크를 통과하여 실제 트래픽을 받기까지의 20초 동안 기존 인스턴스가 단독으로 과부하를 감당해야 했습니다.
      이를 방지하기 위해 70% 시점에 선제적으로 확장을 트리거하여 워밍업 대기 시간을 확보했습니다."

      - "사전에 scale-out 임계치를 설정해두는 자동화된 대응 구조가 이벤트 기반 서비스에서 얼마나 중요한지 직접 경험할 수 있었습니다."
  - key: "Caching"
    title: "인기 게시글 반복 조회를 캐싱 전략으로 응답 속도 안정화"
    image: "/projects/souf-problem.svg"
    imageAlt: "인기 게시글 조회 요청이 데이터베이스에 반복 접근하는 문제"
    problem:
      - "플랫폼 메인 페이지의 인기 게시글의 요청이 증가하면서, 인기 피드 API가 반복적으로 DB에 직접 접근하는 문제가 발생했습니다.\n
      인기 피드는 메인 페이지에서 상시 노출되는 구조를 가졌기 때문에 사용자가 페이지에 접근할 때마다 조회 요청이 발생했습니다.\n
      요청이 들어올 때마다 DB에서 주간 조회 수를 기준으로 상위 6개의 피드를 조회했고, 이 과정에서 DB 커넥션 풀이 지속적으로 사용되었습니다."
    cause:
      - "문제의 핵심은 인기 피드 데이터의 특성과 조회 방식이 어울리지 않는 것에 있었습니다."
      - "인기 피드는 매번 변경되어야 하는 데이터가 아니라, 일정 주기마다 갱신되어도 서비스 품질에 큰 영향을 주지 않는 데이터가 되어야 했지만,
      메인 페이지 접근과 함께 조회 API 요청이 반복적으로 발생하도록 설계한 것이 원인이었습니다."
      - "이런 방식이 지속될 경우, 요청 수가 증가할수록 DB 커넥션 사용량도 함께 증가하는 문제를 초래하고,\n
      사용자에 대한 UX를 크게 저하시킬 수 있는 구조였습니다."
      - "따라서 단순히 쿼리를 개선하는 것보다, 반복적인 DB 접근 자체를 줄이는 캐싱 전략이 더 적합하다고 판단하게 되었습니다."
    solution:
      - "Redis를 직접 제어하는 방식보다, Spring Cache 추상화를 사용하고 실제 저장소는 Redis로 구성하는 방식을 선택하였습니다."
      - "Spring Cache를 선택한 이유는 인기 피드 데이터가 자료구조를 직접 다루는 데이터가 아닌 주간 조회 수 기준으로 집계된 Top6 결과를 저장하고 반환하는
      단순 조회 결과 캐싱에 가까웠기 때문입니다. 따라서 RedisTemplate로 Redis 명령어를 직접 다루기보다 CacheManager를 사용해
      캐싱 의도를 명확히 드러내는 방식이 더 적합하다고 판단했습니다."
      - "이후 인기 피드를 언제 갱신할지 결정했습니다.\n
      Google Analytics 탐색 기능을 통해 사용자 접속률이 가장 낮은 시간이 매주 월요일 자정대임을 확인했습니다.
      그래서 매주 월요일 00시 5분에 스케줄러가 인기 피드를 집계하고 Redis 캐시를 갱신하도록 설계했습니다."
      - "캐싱 전략은 Refresh-ahead를 주전략으로 선택했습니다.\n
      인기 피드는 갱신 시점 외에는 대부분 읽기 요청만 발생하는 데이터였기 때문에, 사용자가 요청한 뒤 캐시를 채우는 방식보다
      스케줄러가 미리 캐시를 갱신해두는 방식이 더 적합했습니다. 다만 Refresh-ahead만으로는 서버 재시작이나 캐시 미스와 같은 예외 상황을 모두 처리하기 어렵다고 판단했습니다.
      캐시가 비어있는 상황에서도 서비스가 정상적으로 운영되어야 했기 때문에 Look-aside를 fallback 전략으로 함께 구성했습니다."
      - "결과적으로 일반적인 상황에서는 스케줄러가 미리 갱신한 Redis 캐시를 통해 DB 접근 없이 인기 피드를 반환하고, 
      캐시가 비어 있는 예외 상황에서는 DB에서 데이터를 조회한 뒤 다시 Redis에 저장하도록 만들었습니다.\n
      이를 통해 DB 커넥션 풀 부담을 줄이면서도, 캐실 유실 상황에 대응할 수 있는 인기 피드 갱신 파이프라인 구조를 만들었습니다."
    result:
      - "평균 응답 시간 512ms → 57ms, 약 88.9% 단축"
      - "메인 페이지 응답 지연 문제 해결"
      - "사용자가 처음 접하게 되는 페이지의 문제를 조기에 발견하여 UX를 개선"
    lesson:
      - "캐싱 전략은 단순히 Redis를 도입하는 문제가 아니라, 데이터의 목적과 갱신 주기를 먼저 이해해야 한다는 점을 배웠습니다."
      - "단순 조회 결과 캐싱에는 Spring Cache 추상화를 활용하고, 다중 인스턴스 환경에서는 Redis를 외부 캐시 저장소로 두는 방식을 적용하며
      Spring Cache와 Redis가 필요한 상황을 명확하게 이해하게 되었습니다."
      - "Refresh-ahead만으로는 캐시 유실 상황을 대처할 수 없다는 문제를 Look-aside를 fallback으로 구성하며, 캐시가 없을 때 어떻게 복구할지 대비 방안을 설계하는 과정에서 어떤 전략이든 예외 상황을 고려해야한다는 것을 경험했습니다."

      - "중복 실행을 방치할 경우, 동일한 데이터를 여러 인스턴스가 동시에 읽고 쓰면서 데이터 정합성이 오염될 가능성이 있으며, DB 부하가 가중이 되는 결과가 이어질 가능성이 매우 높았습니다."
    cause:
      - "해당 문제는 각 서버 인스턴스는 독립적인 프로세스로 동작하기 때문에, 별도의 락을 걸어주지 않는다면 다른 인스턴스의 실행 여부를 알 수 없는 것이 원인이었습니다."
    solution:
      - "문제를 해결하기 위해 저는 Redis 기반 분산락을 선택하기로 했습니다. 하나의 Redis 인스턴스에서 락을 관리하기 때문에 DB 부하 없이 인스턴스 간 락 공유가 가능하며, 추가로 TTL(Time-To-Live) 설정으로 장애 시 자동 해제를 보장할 수 있다는 것이 제 판단이었습니다."
    result:
      - "다중 인스턴스 환경에서 스케줄러가 단일 인스턴스에서만 실행되는 것을 보장하였고, DB 중복 쿼리 및 캐시 불일치 문제를 해소할 수 있었습니다."
      - "분산락 적용 전 로그에 동일 작업이 3번 찍혔으나, 적용 후에 로그에 1번만 찍히는 것을 확인할 수 있었습니다."

    lesson:
      - "정량적 수치는 측정하지 못했으나, 로그 기반으로 단일 실행을 확인하였고 향후에 모니터링 지표 설정의 필요성을 느꼈습니다."
      - "단일 서버 환경에서 정상적으로 동작하던 코드가 다중 인스턴스 환경에서는 전혀 다른 문제를 일으킬 수 있다는 것을 경험했습니다. 확장성을 고려한 설계가 단순히 성능만의 문제가 아니라 데이터 정합성과 직결된다는 점, 그리고 락 전략 선택 시 환경과 트레이드오프를 함께 고려해야 한다는 것을 배웠습니다."

      - "이벤트 기반 트리거: 피드 생성/수정 시 동영상 업로드가 발생하면 EventListener를 통해 AWS SQS로 메시지 전송"
      - "비동기 썸네일 추출: SQS 메시지를 트리거로 ECS가 FFmpeg 컨테이너를 실행하여 썸네일 추출 작업 수행. 업로드 응답과 완전히 분리된 비동기 처리로 API 응답 지연 제거"
      - "피드 조회 개선: 이후 피드 리스트 조회 시 동영상 원본 대신 추출된 썸네일 이미지를 반환하도록 변경"
    result:
      - "피드 리스트 조회 시 동영상 원본 파일 참조를 완전히 제거하여 응답 시간 대폭 단축."

      - "비동기 파이프라인은 단순 성능 개선뿐 아니라 실패 격리와 운영 안정성에도 중요한 설계 선택임을 확인했습니다."

    image: "/projects/souf-result.svg"
    imageAlt: "Google Analytics 경로 탐색 분석과 이탈률 개선 결과"
    problem:
      - "Google Analytics를 통해 메인 페이지와 회원가입 페이지에서 사용자 이탈이 집중적으로 발생하는 것을 확인. 피드 이벤트 운영 중임에도 활성 사용자가 9명에 그쳐 플랫폼 성장에 심각한 병목으로 작용."
    cause:
      - "게릴라 테스트(Guerrilla Testing)를 통해 두 가지 핵심 원인을 파악. 첫째, 메인 페이지에 플랫폼의 핵심 가치를 전달하는 콘텐츠가 부족하여 사용자가 서비스 이용의 필요성을 느끼지 못하고 이탈. 둘째, 회원가입이 단일 페이지에 집중된 복잡한 입력 폼으로 구성되어 진행 상황을 파악하기 어렵고 이탈률이 높았음. 분석 결과를 JIRA Confluence에 정리하여 팀 전체와 공유하고 개선 방향 수립."
    solution:
      - "메인 페이지와 회원가입 두 영역을 분리하여 개선."
      - "메인 페이지 개선 랜딩 화면 상단에 핵심 콘텐츠를 노출하도록 UI 개편하여 사용자가 서비스 가치를 즉시 인지할 수 있도록 구조 변경."
      - "회원가입 플로우 재설계 (백엔드) 단일 폼을 3단계 스텝 구조로 분리하여 사용자가 진행 상황을 명확히 인지하고 완료 기대감을 유지하도록 재설계. 단계별로 유효성 검사 범위를 분리함으로써 API 오류 발생 시 원인 파악과 수정이 용이한 구조로 개선."
      - "소셜 로그인을 병행 도입하여 입력 부담을 최소화. 회원가입 유형(일반/소셜)별 데이터를 sealed class로 명확히 분리하여 컴파일 타임 타입 안전성 확보 및 분기 처리 누락 방지."
    result:
      - "메인 페이지 이탈률 31% → 5% 로 개선."
      - "회원가입 사용자 30명 → 238명 으로 약 693% 증가."
    lesson:
      - "기능 구현 중심의 개발에서 벗어나 Google Analytics 데이터를 직접 분석하고 사용자 행동 기반으로 개선 방향을 도출하는 경험을 했습니다. 백엔드 개발자도 프론트엔드 UX 흐름과 데이터 분석에 대한 기본적인 이해가 있어야 더 나은 API 설계와 서비스 개선이 가능하다는 것을 배웠습니다."
